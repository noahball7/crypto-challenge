Challenge 2 process log
=======================

Purpose
-------

This is a record of the major paths tried while solving `challenge2.txt`.
It is not a proof that every possible variant of each path is false. A "no hit"
means the specific artifact strings, transforms, layouts, KDFs, and AAD choices
tested in this workspace did not produce a valid plaintext.

Personal context
----------------

This was my first time working through this style of multi-step crypto puzzle,
so I kept the process log even when a branch turned out to be wrong. The most
interesting non-final branches were the Project Glasswing/OpenBSD path, the X
profile search that surfaced two Beatles-related tweets, and the Cryptopals
`YELLOW SUBMARINE` check. They were useful because they made the search more
testable, even when they did not decrypt line 1.

Solver provenance
-----------------

GPT Pro was the run that actually found the authenticated plaintext. The full
transcript is in `gpt-pro.md`.

One visible in-flight update from that run captured the right search pressure:

```text
One concrete update: preserving the noncanonical Base64 pad bits yields a
75-byte payload starting with 03, making a versioned AEAD envelope more
plausible than the raw 74-byte decode.

The search is shifting toward T3-Content/skatebench: commit 6058ed9..., PR #5,
hidden/bidirectional Unicode, and the test signatureHash look like the missing
locator surface.

Newly useful detail: the raw skatebench JSON may contain line-separator
characters, so invisible bytes could matter. Direct tre/laser signature-hash
seeds still fail under both leading AEAD parses.
```

That status was useful, but it was not the final decrypt. The final GPT Pro hit
returned to the normal 74-byte Base64 decode, applied byte inversion, and parsed
the result as `v1 | salt16 | iv12 | tag16 | ciphertext29`.

Final solved path
-----------------

The final solution was:

```text
Base64(line 1) -> invert each byte with byte ^ 0xff
-> parse v1 | salt16 | iv12 | tag16 | ciphertext29
-> key = SHA256("A drum break might shatter it!" || ":" || raw_salt)
-> AES-256-GCM, no AAD
-> It's all in the mind, y'know?
```

The reusable verifier is:

```sh
node scripts/verify_challenge2_solution.mjs
```

Final pivot: inversion and tag-before-ciphertext
------------------------------------------------

What changed:

- GPT Pro tried the operation hinted by `switch` / inversion against the normal
  Base64-decoded bytes.
- Instead of treating the decoded bytes as direct ciphertext/signature data,
  invert every byte:

```text
raw[0..] ^ 0xff
```

Why it made sense after the fact:

- Raw first bytes:

```text
fe b4 ...
```

- Inverted first bytes:

```text
01 4b ...
```

- `01` works as a version byte.
- The remaining bytes fit exactly:

```text
salt16 | iv12 | tag16 | ciphertext29
```

The final parse:

```text
v    = 01
salt = 4b97b65593e4b667d928333b5fc8d1d7
iv   = e38fac4d6b4066d044d2d433
tag  = 8a89b0472262537287b7cb40552be4a9
ct   = fadf6ffa063fda8b8f9dbfb2e7e3a635d635a81dcc210b5682ed2515b7
```

The exact seed:

```text
A drum break might shatter it!
```

The exact KDF:

```text
SHA256(seed || ":" || raw_salt)
```

The result:

```text
It's all in the mind, y'know?
```

The exact GPT Pro hit shape was:

```text
HIT not-v1-tagct A drum break might shatter it!  b"It's all in the mind, y'know?"
```

Process takeaways
-----------------

- The right source string was already in the first strong clue path.
- Broad semantic expansion was less useful than exact phrase preservation.
- AES-GCM authentication was the best filter: plausible candidates without a
  valid tag were rejected.
- Challenge 1 gave the right KDF shape, but not the right AAD behavior or
  envelope layout.
- The noncanonical Base64 / skatebench / hidden-Unicode lead was useful search
  pressure, but the authenticated hit came from the inverted 74-byte decode.
- The OpenBSD/signify branch was false as a signature theory, but the failed
  `fe b4` marker became meaningful once byte inversion was tried.
